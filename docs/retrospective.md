# Project Retrospective

This document looks back on the engineering journey of building NeuroFlow across twenty structured tasks. It covers the hardest technical challenges, design decisions we would reconsider, and lessons learned about building production AI systems.

The most difficult technical challenge we faced was during our testing and continuous integration phase. We encountered a major roadblock when our requirements text file was accidentally overwritten by a pip freeze command. This corrupted our dependency tree and completely broke our test suite. We had to manually restore the production dependencies. Another major challenge was writing unit tests for our circuit breaker. The circuit breaker used an asynchronous context manager to connect directly to the database. Our initial synchronous tests crashed completely. We had to learn how to use asynchronous mocks to isolate the circuit breaker from the real database and execute the tests properly. We also fought with strict static typing where we had to resolve tricky circular imports between our message and pipeline modules.

Looking back at our design decisions we would probably rethink our decision to use a single database for both the message broker and the caching layer. While it is very fast and convenient it became clear during load testing that a single instance could become a bottleneck if the cache gets overwhelmed by large prompt generations while simultaneously trying to manage a deep ingestion queue. Decoupling the message broker into a dedicated message queue would have given us more resilience at scale even if it added infrastructure complexity early on.

Building this system taught me a lot of things that simple tutorials never mention. Tutorials usually show you the happy path where you just plug a key into a library and everything works perfectly. In reality production systems are mostly about what happens when things fail. You have to build circuit breakers for when the provider goes down and rate limiters for when you run out of tokens. You also have to deal with prompt injection attacks and validate every piece of input because users will try to break your system. I learned that an AI feature is only ten percent of the work while the other ninety percent is infrastructure security and observability.

The metric improvement sprint was a real eye opener. I originally thought that improving search quality just meant switching to a bigger embedding model. I learned that you can achieve massive gains just by tuning simple parameters like chunk sizes and search depths. By aggressively testing our chunk size and caching identical queries we slashed our latency and dramatically improved context precision without writing any complex new features. It showed me that measuring your current state and making small data driven tweaks is much more effective than guessing.
